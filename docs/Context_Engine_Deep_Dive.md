# 战役1 工程深挖：Context Engine 的内部器官 (Deep Dive)

如果 Captain 瞎了，那就意味着 Context Engine 必须是一个算力极高的“最强外脑”。在这里，我们不容许糊弄。所有的复杂状态管理，必须在这个黑盒中极其优雅地自我闭环。

## 1. 记忆分级与“降维熔炼”机制

Context Engine 不能只是简单地把历史记录拼接 (Concat) 后丢给 Captain。当遭遇超长文本或多轮扯皮时，这会立刻引起 Token 溢出。

**工程硬规则：**
必须构建一套独立于主回复周期的“后台降维队列 (Background Reduction Queue)”。
*   **L1 (短期极速内存)**：仅保留最近 3 轮未被降维的裸数据（Raw Token）。
*   **L2 (长期脱水法典)**：每当 L1 蓄水池满 3 轮，引擎在后台静默唤起一个廉价、高并发的小模型 (如 gemini-flash 或 glm-4-flash)，将这 3 轮扯皮强行总结成一两句极简的“核心事实 (Core Directives)”，并入 L2 档案（如 markdown 结构），然后清空 L1。
*   **最后组装**：给 Captain 的子弹 = `L2的几行真理大纲 + 最新1次的短期对话 + 当前任务`。

## 2. 快照的原子性 (Atomic Snapshot)

多通道、多 Agent 并发访问 Context Engine 时，如果正在一边写记忆一边读切片，会发生状态雪崩（像 OpenClaw v2026.3.8 里抢破头的那些补丁一样）。

**工程硬规则：构建写时复制 (COW, Copy-On-Write) 读档法**
*   当 Captain 请求一发子弹（Snapshot）时，Context Engine 不允许返回任何引用或动态游标。
*   它必须根据请求时的瞬间（Timestamp），强制生成一份带有 `Hash(Content + Timestamp)` 的静态 JSON 物理文件。Captain 在沙箱里只对着这个不可变的静态实体切片发力。
*   如果执行过程中发生了其它打断，这个切片的 Hash 已经锁定，不会被外部干扰（绝对防越权交叉）。

## 3. 对话恢复原点的硬熔断 (The Hard Reset Protocol)

所有的历史记忆都会随着时间变成剧毒包袱（如“忘了前置规则”、“出现幻觉”）。这就是以前系统越来越卡的根源。

**工程硬规则：强迫式遗忘点 (Forced Reset Anchors)**
*   Context Engine 必须能够侦测任务的终结或重大流转（例如“归档结案”、“进入另一主题”）。
*   一旦触碰锚点，引擎将当场引爆 L1 和 L2 的全部记录，只用单向向上的提取机制抽出一行（如：写下一篇新法典或总结），然后所有的历史状态物理 Flush。
*   不再提供“回档到 10 分钟前”这种功能。不要妥协。要回到原点，就必须从彻底打碎一切重新堆叠。
