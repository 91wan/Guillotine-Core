# 黑盒监狱与静默清洗区方案 (The Guillotine Purifier)

## 🎯 核心痛点与目的
在原生系统中，由于缺乏沙箱出口的强力管束，经常导致诸如 `<think>` 标签泄漏、`NO_REPLY` 静默残片跳出，甚至包含冗余的报错堆栈被打印到用户的 UI 上。这些暴露面不仅显得极不专业，还在不断逼迫前端同学去写恶心的 `if...else` 过滤补丁。

本方案的目标是：**建造一座只进不出的黑盒监狱。所有在狱内（子 Agent 或子循环）产生的脏活累活和思维脚手架，在跨出牢门（即返回给 UI/Captain 接收端）的瞬间，必须被剥得干干净净。**

## 🔪 机制拆解

### 1. 物理屏蔽：沙箱内外绝对隔离 (The Air Gap)
*   **绝对阻断**：主控进程（Captain）与子进程（Sub-Agent / 沙箱）之间，禁止使用事件流 (SSE/Delta Stream) 来传递中间过程。
*   **黑盒执行**：子进程只能把思考过程（包括调用什么 Tools、报了什么 Warning、推翻了什么想法）写在监狱内壁的一块黑板（临时日志文件 `execution.trace`）上。
*   **只交结果**：执行完毕，只允许向铁门外的世界传递一个极其简单的数据结构：`{"FinalOutput": "..."}`。这叫做“完工单”。

### 2. 出口强力清洗 (Left-Shift Cleansing)
清洗的职责必须在**生成源头**进行（左移），绝不要指望接收端去帮你擦屁股。
*   **强制脱壳**：在生成完工单前，必须经过一层暴力的正则/解析剥离管线，自动剃除任何包含在 `<think>...</think>` 以及隐式推演段落的冗余文字。
*   **碎片抹杀**：对 `NO_REPLY`、`{ ...ToolCall... }` 此类机器指令残留做精确匹配拦截。一旦命中，该完工单直接置为空字符串 `""`，而不是向外暴露出残缺符号。

### 3. 法医归档 (阅后即焚)
*   那块写满推演的黑板 (`execution.trace`)，在完工单递出去的同一时刻，由沙箱主管直接进行物理擦除（`rm` 或强制覆盖）。
*   Captain 收到的只是一句神启般的结论。
