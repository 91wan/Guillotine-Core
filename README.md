# Guillotine Core for OpenClaw 🔪
> The Zero-State, Zero-Tolerance, Hard-Isolated Architecture.

如果你的 AI Agent 经常因为会话历史太长而 Token 爆仓、在 UI 界面上呕吐出 `<think>` 标签和堆栈报错、或者因为一个外部大模型的 `429 Rate Limit` 就全盘崩溃——**那是因为你的架构太懦弱了。**

**Guillotine Core (断头台核心)** 是一套旨在将 OpenClaw 等 Agent 框架重塑为“绝对无状态杀戮机器”的外部挂载组件/微服务设计法典。

我们绝不妥协于打补丁式的修复。我们选择物理切除。

## 🎯 核心法典与三大武器 (The Three Guillotines)

### 1. Context Guillotine (记忆斩断与强制降维)
原生系统试图把对话历史无限塞进内存。**我们选择斩断记忆神经。**
*   **机制**: Captain (主路由) 绝对致盲，不可持存任何历史。所有记忆交给独立的后台引擎处理。每 3 轮短兵相接，后台强制召唤廉价算力进行物理压缩与脱水，提纯为 2 行绝对法则。
*   **特性**: `COW (写时复制)` 原子快照。Captain 拿到的永远是一个带有 Hash 锁死、拒绝任何指针篡改的静态 JSON 任务单。

### 2. Gatekeeper Proxy (叹息之墙网关)
让应用层去维护 API 拥堵和报错重试是架构的耻辱。
*   **机制**: 剥夺沙箱的外交侧网络直连权。所有对外界大模型的请求必须打向本地的强权网关。
*   **特性**: 隐瞒与擦屁股。遇到 `429 限流` 或 `502 过载`，网关在内部指数退避默默扛下，甚至偷偷更换连线模型，最终只向内部微服务返回冷酷的 `200 OK` 或是 `503 彻底断头`。内部系统不再拥有等待、重试的概念。

### 3. Purifier Valve (静默清洗闸门)
UI 前端不应该去负责过滤那些恶心的机器脚手架代码。
*   **机制**: 黑盒监狱式的沙箱模型。所有的思考、报错、日志，全部留在黑盒内的黑板上。
*   **特性**: 左移清洗与阅后即焚。在交出执行结果跨出监狱大门的瞬间，被正则暴君强制刮骨，剃除所有的 `<think>`、`NO_REPLY` 和堆栈血迹，只留下干净的结构体。内部日志当场物理 `rm` 销毁。

## 📁 概念验证 (Proof of Concept)
本项目包含我们实装的纯物理隔离打样代码，你可以直接在本地靶场体验这套冷血运行机制：

*   `poc/captain_runtime.sh` - 无状态核心调度体验
*   `poc/context_engine.py` - 每 3 轮自动降维脱水与 COW 快照生成器
*   `poc/gatekeeper_proxy.py` - 对内报喜不报忧、拦截 429 报错的叹息之墙代理
*   `poc/purifier_sandbox.py` - 左移的强制正则隔离刀与阅后即焚模拟

*(这不仅是一套方案，更是带给平庸架构的一场葬礼。)*
